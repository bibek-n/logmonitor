# QA Testing Management

Test suites, test cases, test runs/execution, and bug tracking, integrated into the existing
Next.js + MSSQL app — not a separate application. Everything lives under
`src/app/api/admin/qa/**` (API), `src/app/dashboard/qa/**` (pages), `src/components/qa/**`
(UI), and `src/lib/qa*.ts` (shared logic).

## Install

```
npm run migrate:qa-testing
npm run migrate:qa-workflow-gates
```

The second migration adds the requirement/review/deployment/approval/release columns
described in "Workflow gates" below — run it once after the base migration; both are
idempotent and safe to re-run.

Idempotent — safe to re-run. Creates 14 tables, their indexes, and seeds four roles
(`QA Manager`, `Tester`, `Developer`, `Viewer`) with default `RolePermissions` grants (see
below). No new environment variables. No separate migration is needed for existing
installs to pick up the new sidebar group — it appears automatically for any user whose
role has the `qa_view` permission.

## Database

New tables (all `Id INT IDENTITY(1,1) PRIMARY KEY`, named FK constraints, no
`ON DELETE CASCADE` — matches this app's existing convention everywhere else):

| Table | Purpose |
|---|---|
| `QaProjects`, `QaModules`, `QaReleases` | Organizational lookups. New — this app had no project/module/release concept before this module. |
| `QaTestSuites` | Groups test cases within a project/module. `Status`: `Active` / `Archived`. |
| `QaTestCases` | `TestCaseNumber` (e.g. `TC-00042`, generated by the app, not the DB — see "Reference numbers" below). `Status`: `Draft` / `Ready` / `Approved` / `Deprecated` / `Archived`. |
| `QaTestCaseSteps`, `QaTestCaseTags` | Child rows — one-to-many per test case. |
| `QaTestRuns` | `TestRunNumber` (`TR-00007`). `Status`: `Planned` / `In Progress` / `Paused` / `Completed` / `Cancelled`. |
| `QaTestRunCases` | Join table: which test cases are in a run, and who's assigned (`AssignedUserId`, nullable). Unique on `(TestRunId, TestCaseId)`. |
| `QaTestExecutions` | **Append-only** — one row per execution attempt, never overwritten. The newest row per `TestRunCaseId` (by `ExecutedAt`) is the current result; older rows are history. |
| `QaBugs` | `BugNumber` (`BUG-00019`). `Status`: `New` / `Open` / `In Progress` / `Resolved` / `Ready for Retest` / `Verified` / `Closed` / `Rejected` / `Duplicate` / `Reopened`. `ResolvedAt` is auto-stamped when `Status` enters `Resolved`/`Verified`/`Closed`, and cleared when it leaves that set (e.g. `Reopened`) — see `BUG_RESOLVED_STATUSES` in `src/lib/qaShared.ts`. |
| `QaAttachments` | Polymorphic (`EntityType` ∈ `TestCase` / `TestExecution` / `Bug`, `EntityId`). Files stored outside the web root (`qa-attachments/`), served only via authenticated `GET /api/admin/qa/attachments/[id]`. |
| `QaComments` | Created by the schema migration but **has no API or UI built** — out of scope for this pass. The table exists so a future comment-thread feature doesn't need a schema change. |
| `QaActivityLogs` | This module's own detailed audit trail (`EntityType`/`EntityId`/`Action`/`PreviousValue`/`NewValue`/`UserId`/`IpAddress`) — written in addition to, not instead of, the app-wide `AdminAuditLog` every mutation also logs to via `logAdminAction()`. |

**Reference numbers** (`TC-00042`, `TR-00007`, `BUG-00019`) are generated by
`withReferenceNumber()` in `src/lib/qaReferenceNumbers.ts` — the only place in this app that
uses a real SQL transaction (`SELECT MAX(...) WITH (UPDLOCK, HOLDLOCK)` + the insert, in one
transaction), specifically to stay collision-free under concurrent requests. Verified under
real concurrent load in `scripts/test-qa-integration.ts`.

**No hard delete** for Projects, Test Suites, or Test Cases — only archive (`Status`/`IsActive`
toggle), because `QaTestRunCases`/`QaBugs` reference them and this app never uses
`ON DELETE CASCADE`. Test Runs and Bugs don't need archiving since their own `Status` field
already models a terminal state.

## Permissions

Nine new keys, added to the existing (previously unenforced) `RolePermissions` mechanism —
see `src/lib/permissionKeys.ts`. `requireQaPermission(key)` (API routes) and `getQaAccess()`
(Server Component pages, resolves every key at once so a page can show/hide several buttons
with one DB round trip) are in `src/lib/requireQaPermission.ts`. **Admin always bypasses
every check**, same superuser convention `requireAdmin()` uses everywhere else in this app.

| Key | Meaning |
|---|---|
| `qa_view` | View suites/cases/runs/bugs. Also gates whether the "QA Testing" sidebar group is shown at all. |
| `qa_create` | Create projects, suites, test cases; clone/import test cases. |
| `qa_edit` | Edit suites and test cases; attach files to a test case. |
| `qa_delete` | Archive suites/test cases. |
| `qa_execute` | Submit/update execution results; attach files to an execution; see "my assignments". |
| `qa_manage_runs` | Create/edit test runs; add/remove/reassign cases in a run. |
| `qa_manage_bugs` | File bugs, edit bugs, change bug status/assignment; attach files to a bug. |
| `qa_view_reports` | QA Reports page, dashboard KPI/trend endpoints, CSV export. |
| `qa_admin` | Reserved — not currently granted to any seeded role or checked anywhere in this module's code. |

Default role grants (seeded by the migration; editable afterward in the existing Roles admin
UI like any other role):

| Role | Grants |
|---|---|
| `QA Manager` | `qa_view` `qa_create` `qa_edit` `qa_delete` `qa_execute` `qa_manage_runs` `qa_manage_bugs` `qa_view_reports` |
| `Tester` | `qa_view` `qa_execute` `qa_manage_bugs` |
| `Developer` | `qa_view` `qa_manage_bugs` |
| `Viewer` | `qa_view` `qa_view_reports` |
| `Admin` | Everything, unconditionally (bypass, no grant rows needed) |

## Workflow gates

Added by `npm run migrate:qa-workflow-gates`, mapping the module onto the full requirement →
release pipeline (suites/cases/runs/execute/bugs/fix/retest already existed from Phase 3/4):

| Step | Where it lives | How |
|---|---|---|
| Requirement Received | `QaTestSuites.RequirementRef` | Free-text reference (ticket link, spec name) set when creating/editing a suite — no ticket-system integration, just traceability. |
| Review Test Cases | `QaTestCases.ReviewedByUserId` / `ReviewedAt` | Explicit "Mark Reviewed" action on the test case detail page, distinct from `Status` — a case can be `Ready` without a human having actually reviewed it. Editing other fields doesn't auto-clear a review; a reviewer clears it manually via "clear" if a change warrants re-review. |
| Deploy Application to QA | `QaTestRuns.DeployedBuildVersion` / `DeployedAt` | Recorded on the test run detail page — a build version stamps the deployment event. Can be re-recorded if the run gets redeployed mid-cycle. |
| Regression Testing | `QaTestRuns.RunTypeId` → `QaTestRunTypes` | Set at run creation from a DB-backed lookup, not a fixed enum — see "Test run types" below. A run whose type is "Regression Test" is treated as a regression pass in the workflow stepper. |
| QA Approval | `QaTestRuns.QaApprovedByUserId` / `QaApprovedAt` | `PATCH /test-runs/[id]` with `{ qaApproved: true }`. **Server-enforced gate**: only allowed when the run's `Status = 'Completed'` AND it has zero open Critical/High severity bugs still linked via `TestRunId`. Never trusts a client-sent approval flag. |
| Production Release | `QaReleases.Status = 'Released'`, `ReleasedByUserId` / `ReleasedAt` | `PATCH /releases/[id]` with `{ status: 'Released' }`. **Server-enforced gate**: requires at least one test run linked to the release (`QaTestRuns.ReleaseId`) to be both `Completed` and QA-approved. A release page (`/dashboard/qa/releases`) lists every release and its linked runs — this didn't exist before this pass. |

List endpoints (`test-cases`, `test-suites`, `test-runs`, `bugs`) all support
`?page=&pageSize=&sortBy=&sortDir=&search=` plus entity-specific filters (see
`buildTestCaseFilters`/`buildTestRunFilters`/`buildBugFilters` in `src/lib/qaShared.ts`).

## Test run types

Added by `npm run migrate:qa-test-run-types`. Replaces the earlier free-text
`QaTestRuns.RunType` (`New Feature`/`Regression`/`Smoke`/`Full Regression`, from the workflow-
gates pass above) with a proper lookup table plus a many-to-many link to test cases, so
creating a run of a given type can auto-load the matching cases instead of picking every case
by hand:

- `QaTestRunTypes` — 42 seeded types across three migrations: the original 8
  (`migrate:qa-test-run-types`: Smoke/Regression/Release/Security/Browser Compatibility/Mobile/
  Production Verification/Custom), 25 advanced/non-functional types
  (`migrate:qa-advanced-run-types`: Load/Stress/Spike/Endurance/Scalability/Failover/Recovery/
  Backup and Restore/Database Integrity/API Reliability/Network Failure/Session and
  Concurrency/Permission and Role/Data Migration/Upgrade and Downgrade/Installation/
  Configuration/Compatibility/Accessibility/Usability/Localization/Audit and Logging/
  Notification/File Upload Security/Chaos), and 9 more from a supplied "standard list"
  (`migrate:qa-run-types-standard`: Functional/Sanity/Integration/System/User Acceptance/API/
  Database/Upgrade/File Upload — kept distinct from their closest existing cousin rather than
  merged, since they were named separately). Admin-managed only via direct DB edit today — no
  CRUD UI, since the set rarely changes.
- `QaTestCaseRunTypes` — junction table (`TestCaseId`, `RunTypeId`). A test case can carry any
  number of run types via the "Run Types" checkbox group on the test case form, separate from
  the single `TestType` field (Functional/Regression/Smoke/.../Security — an orthogonal
  classification of *what kind* of test it is, not *which runs* it belongs to).
- `QaTestRuns.RunTypeId` — set at run creation. The old `RunType` VARCHAR(20) column is still
  in the table (unused by current code, backfilled once at migration time) rather than
  dropped, consistent with this app's no-destructive-schema-change convention.

**Auto-load flow**: the "New Test Run" form's Run Type picker fetches `GET /test-run-types`;
once a project and run type are both picked, an "Auto-load matching test cases" checkbox (on
by default) fetches `GET /test-cases?projectId=&runTypeId=` and pre-selects every match, which
the user can then narrow down before submitting `POST /test-runs` with the final
`testCaseIds`. Unchecking auto-load (or picking "Custom Test") leaves the picker empty for a
fully manual run — cases can always be added afterward from the run detail page regardless.

## Structural core: Requirements, Test Plans, Milestones, Environments, Builds

Added by `npm run migrate:qa-structural-core` — the foundational layer beneath a TestRail/
Zephyr-style tool, built on top of the existing suite/case/run/bug machinery rather than
replacing any of it. All five follow the same no-hard-delete, Status-lifecycle convention as
every other QA entity, and none introduce new permission keys (all gated by the existing
`qa_view`/`qa_create`/`qa_edit`).

- **Requirements** (`QaRequirements`, ref numbers `REQ-00001`) — Title/Description/Category
  (free text)/Priority (reuses the Low-Critical scale)/Status (New → Approved → Implemented →
  Verified, or Deprecated). Linked to test cases many-to-many via `QaRequirementTestCases`
  (`/dashboard/qa/requirements/[id]`'s "Link Test Cases" action). The **Traceability Matrix**
  (`/dashboard/qa/requirements/traceability`) joins every requirement in a project to its
  linked cases and each case's latest execution result, computing a coverage % per requirement
  (Passed executions ÷ linked cases) — read in full per project, not paginated.
- **Test Plans** (`QaTestPlans`, ref numbers `TP-00001`) — wraps multiple Test Runs via
  `QaTestPlanRuns`, optionally tied to a Release. The detail page aggregates pass/executed
  totals across every linked run (same "latest execution per run-case" query as the test run
  detail page, just summed).
- **Milestones** (`QaMilestones`) — Sprint or Release Milestone type, optional due date and
  Release link, wraps multiple Test Plans via `QaMilestoneTestPlans`. Progress is simply
  completed-plans ÷ linked-plans.
- **Environments** (`QaEnvironments`) — project-scoped DEV/QA/UAT/STAGING/PRODUCTION-style
  references (Name/ApiUrl/DatabaseInfo/BuildVersion/ConfigNotes), all free text and
  **deliberately descriptive-only — no real credentials belong here**, since this is a public
  repo. Simple list + create/edit modal, no separate detail page.
- **Builds** (`QaBuilds`) — BuildNumber/GitCommit/Branch/DeploymentDate/Status
  (Pending/Deployed/Failed/Rolled Back), optionally linked to a Release and an Environment.
  Same list + modal treatment as Environments.

Both Environments and Builds are wired into Test Run creation: `QaTestRuns.EnvironmentId` and
`QaTestRuns.BuildId` were added as nullable columns alongside the pre-existing free-text
`Environment`/`DeployedBuildVersion` fields (same additive, non-destructive pattern used for
`RunTypeId`) — the "New Test Run" modal shows an optional managed dropdown for each, populated
per-project, in addition to the always-available free-text inputs.

The "QA Testing" sidebar group is the only permission-gated nav group in this app — every
other group is visible to any authenticated user regardless of role. Individual buttons
within a page (Create/Edit/Archive/Manage Runs/etc.) are separately hidden per the visiting
user's specific grants via `getQaAccess()`, but the API is what actually enforces
permission — hiding a button is a UI convenience, not a security boundary.

`src/components/qa/QaTable.tsx` is the shared paginated/sortable/column-visibility/
bulk-select table every list page uses. `src/components/qa/useQaList.ts` is the client hook
that drives it against the REST API above (list pages fetch client-side rather than via
server-side DB queries, specifically so filtering/sorting/pagination logic isn't duplicated
between the page and the API route that already implements it).

Projects have no dedicated management page (not in the original page list) — they're created
inline via the "+ New" quick-add in `src/components/qa/ProjectSelect.tsx`, used everywhere a
form needs a project.

## Testing

- `npm test` — Vitest unit tests (CSV parsing, filter/validation logic, permission
  resolution with mocked auth/DB). No DB connection needed.
- `npm run test:qa-integration` — real-DB integration test
  (`scripts/test-qa-integration.ts`): reference-number concurrency, execution history
  ordering, FK/unique-constraint enforcement, bug `ResolvedAt` semantics. Self-cleaning —
  tags every row it creates and deletes them (plus a leftover sweep) in a `finally` block.
  Safe to run against production; there's no separate test database for this app.
- `e2e/qa-critical-workflow.spec.ts` — Playwright, covers the full workflow (create suite →
  case with steps → run → assign tester → execute → mark failed → file bug → resolve/reopen
  → complete run → view dashboard). Written but not yet run against production — see
  `e2e/README.md` for what authenticating it requires and why that step was left to be done
  manually.

## Known gaps

- `QaComments` has a table but no API/UI.
- CSV import/export exists only for Test Cases.
- No project-level access restriction (e.g. "Tester X can only see Project Y") — every QA
  permission is global once granted, same as every other role in this app.
- Explicitly out of scope for the "structural core" pass (deferred per an explicit prioritization
  decision, not an oversight): live CI/CD integration (GitHub Actions/Jenkins/GitLab/Azure
  DevOps webhook ingestion), Selenium/Playwright/Cypress automation result import, an in-app
  API-testing tool (Postman-style collections), Slack/Teams notifications (email is feasible via
  the app's existing SMTP settings but not yet wired to QA events), a unified file repository
  browser, global search, and a Kanban board / calendar / activity timeline UI layer.
